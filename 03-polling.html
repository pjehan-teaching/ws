Polling

Principe du polling

Le polling (ou interrogation périodique) consiste à envoyer des requêtes HTTP au serveur à intervalles réguliers pour vérifier si de nouvelles données sont disponibles.

C'est la solution la plus simple à implémenter, mais aussi la moins efficace.

Fonctionnement

  1. Le client envoie une requête toutes les N secondes
  2. Le serveur répond immédiatement (avec ou sans nouvelles données)
  3. Le client attend l'intervalle suivant puis recommence

La latence maximale est égale à l'intervalle de polling.

Côté serveur (TypeScript)

import express from 'express';
import path from 'path';

const app = express();
const messages: { id: number; text: string; author: string }[] = [];

app.use(express.json());
app.use(express.static(path.join(import.meta.dirname, '../client')));

app.get('/api/messages', (req, res) => {
  res.json(messages);
});

app.post('/api/messages', (req, res) => {
  const message = { id: Date.now(), ...req.body };
  messages.push(message);
  res.status(201).json(message);
});

app.listen(3000);

Côté client (TypeScript)

Ce code est enregistré dans src/client/main.ts :

async function fetchMessages(): Promise<void> {
  const response = await fetch('/api/messages');
  const messages = await response.json();

  const container = document.getElementById('messages')!;
  container.innerHTML = messages
    .map((m: any) => `<p><strong>${m.author}</strong> : ${m.text}</p>`)
    .join('');
}

// Polling toutes les 2 secondes
setInterval(fetchMessages, 2000);
fetchMessages();

Ajouter un message via curl

Pour tester l'endpoint POST /api/messages sans passer par l'interface, on peut utiliser curl :

curl -X POST http://localhost:3000/api/messages \
  -H "Content-Type: application/json" \
  -d '{"author": "Alice", "text": "Bonjour depuis curl !"}'

Le message apparaît dans le navigateur au prochain cycle de polling (dans les 2 secondes).

D'autres outils permettent d'envoyer ces requêtes sans ligne de commande :

  • Postman / Insomnia : interfaces graphiques complètes pour construire et sauvegarder des requêtes
  • Thunder Client / REST Client : extensions VS Code, requêtes définies directement dans un fichier .http

Analyse du trafic réseau

Ouvrez les DevTools (F12) et observez l'onglet Network.

  • Chaque requête génère des headers HTTP (plusieurs centaines d'octets)
  • La plupart des réponses sont vides ou identiques à la précédente
  • Avec 100 clients et un intervalle de 2s : 50 requêtes/seconde vers le serveur

Analyse critique

Aspect Évaluation
Latence Élevée (dépend de l'intervalle)
Charge serveur Élevée (requêtes inutiles)
Bande passante Gaspillée (headers répétés)
Simplicité Très simple à implémenter
Compatibilité Universelle (simple HTTP)
MDN : Fetch API

Cas d'usage appropriés

Le polling reste pertinent lorsque :

  • Les données changent peu fréquemment (ex : toutes les minutes)
  • Une latence élevée est acceptable
  • Le nombre de clients est faible
  • La simplicité d'implémentation prime sur la performance

Exemples : vérification de l'état d'une commande, rafraîchissement de météo.
