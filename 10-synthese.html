Synthèse & Évaluation

Récapitulatif des techniques

Technique Avantages Inconvénients
Polling Très simple, compatible partout Latence élevée, charge serveur inutile
Long Polling Latence réduite, HTTP standard Connexions longues, gestion complexe
SSE Natif, reconnexion auto, simple Unidirectionnel (serveur → client)
WebSocket Bidirectionnel, faible latence, binaire Nécessite un serveur compatible
Socket.io Fonctionnalités avancées, fallback Overhead, protocole propriétaire

Choisir la bonne technique

La technique à choisir dépend de vos besoins :

  • Polling : données rarement mises à jour, pas de contrainte de latence
  • Long Polling : notifications ponctuelles, environnement restreint (pas de WebSocket)
  • SSE : flux de données serveur → client uniquement (logs, métriques, feeds)
  • WebSocket natif : communication bidirectionnelle avec contrôle total, faible overhead
  • Socket.io : applications complexes avec rooms, reconnexion, multi-transport

Arbre de décision

Besoin de communication en temps réel ?
│
├── Unidirectionnel (serveur → client uniquement) ?
│   ├── Oui → Server-Sent Events (SSE)
│   └── Non → Communication bidirectionnelle
│
└── Bidirectionnel ?
    ├── Simplicité maximale requise ?
    │   └── Oui → Polling simple
    │
    ├── Fonctionnalités avancées (rooms, reconnexion, auth) ?
    │   └── Oui → Socket.io
    │
    └── Contrôle total, performance maximale ?
        └── Oui → WebSocket natif (bibliothèque ws)

Évolution technologique

Période Technologie dominante
2000–2006 Polling (AJAX)
2006–2011 Long polling (Comet)
2011–présent WebSocket (RFC 6455)
2012–présent SSE (EventSource API)
2014–présent Socket.io (abstraction WebSocket)
2022–présent WebTransport (HTTP/3, QUIC)

Et demain ? WebTransport

WebTransport est une API expérimentale basée sur HTTP/3 et QUIC qui promet :

  • Latence encore plus faible que WebSocket (pas de blocage en tête de file)
  • Multiplexage de flux indépendants
  • Support des datagrammes (UDP-like) pour les jeux
  • Meilleure gestion de la perte de paquets

Évaluation pratique

  1. Implémenter une fonctionnalité temps réel de votre choix parmi :
    • Un système de quiz en direct multijoueur
    • Un tableau blanc collaboratif
    • Un outil de gestion de projet avec notifications
    • Un jeu multijoueur simple (morpion, puissance 4...)
  2. Justifier le choix de la technologie utilisée (Polling, SSE, WebSocket, Socket.io)
  3. Le code TypeScript doit être correctement typé
  4. Documenter les choix d'architecture
