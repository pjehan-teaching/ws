Long Polling

Principe du long polling

Le long polling est une amélioration du polling classique : le serveur ne répond pas immédiatement à la requête, mais la maintient ouverte jusqu'à ce qu'une nouvelle donnée soit disponible.

Dès que le client reçoit une réponse, il envoie immédiatement une nouvelle requête.

Fonctionnement

  1. Le client envoie une requête au serveur
  2. Le serveur bloque la réponse jusqu'à avoir des données
  3. Dès qu'une donnée arrive, le serveur répond
  4. Le client reçoit la réponse et envoie aussitôt une nouvelle requête

File d'attente côté serveur

import express, { Request, Response } from 'express';

const app = express();
app.use(express.json());

export type Message = { id: number; text: string; author: string };
type Subscriber = (data: Message) => void;
const subscribers: Subscriber[] = [];
const messages: Message[] = [];

function notifySubscribers(data: Message): void {
  const pending = subscribers.splice(0); // Copier la liste et vider le tableau
  pending.forEach(resolve => resolve(data));
}

function waitForMessage(): Promise<Message> {
  return new Promise(resolve => {
    subscribers.push(resolve);
  });
}

Endpoint long polling

app.get('/api/long-poll', async (req: Request, res: Response) => {
  const TIMEOUT = 30_000; // 30 secondes max

  const timeoutPromise = new Promise<null>(resolve =>
    setTimeout(() => resolve(null), TIMEOUT)
  );

  // Attendre soit un nouveau message, soit le timeout
  const data = await Promise.race([waitForMessage(), timeoutPromise]);

  if (data === null) {
    res.status(204).end(); // Pas de nouvelles données
  } else {
    res.json(data);
  }
});

app.post('/api/messages', (req: Request, res: Response) => {
  const message: Message = { id: Date.now(), ...req.body };
  messages.push(message);
  notifySubscribers(message);
  res.status(201).json(message);
});

Côté client : reconnexion automatique

async function listenForMessages(): Promise<void> {
  try {
    const response = await fetch('/api/long-poll');

    if (response.status === 200) {
      const message = await response.json();
      displayMessage(message);
    }
    // Immédiatement relancer une nouvelle requête
    listenForMessages();
  } catch (error) {
    // Attendre avant de réessayer en cas d'erreur réseau
    console.error('Erreur réseau, reconnexion dans 3s...');
    setTimeout(listenForMessages, 3000);
  }
}

Gestion du timeout côté client

async function listenForMessages(): Promise<void> {
  try {
    const controller = new AbortController();
    const timeoutId = setTimeout(() => controller.abort(), 35_000);

    const response = await fetch('/api/long-poll', {
      signal: controller.signal
    });
    clearTimeout(timeoutId);

    if (response.status === 200) {
      const message = await response.json();
      displayMessage(message);
    }

    listenForMessages();
  } catch (error) {
    if ((error as Error).name !== 'AbortError') {
      setTimeout(listenForMessages, 3000);
    }
  }
}

Connexions ouvertes Courtes Longues (bloquantes)
Complexité serveur Faible Moyenne
Scalabilité Mauvaise Meilleure

Limites du long polling

  • Les connexions ouvertes consomment des ressources serveur (mémoire, descripteurs de fichiers)
  • La gestion des timeouts et reconnexions ajoute de la complexité
  • Les proxies et load balancers peuvent couper les connexions trop longues
  • Problème des messages perdus pendant la reconnexion
